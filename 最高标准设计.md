# kvspace-rdma 最高标准设计

> **范围**：本文档覆盖 server。SDK/client 在 `kvspace-cpp` 中（`kvspace::Client` 接口）。
>
> **接口契约**：与 `kvlang/internal/kvspace/kvspace.go` 完全一致的 12 个方法；
> server 是这个契约的通用传输实现，不增减语义。
>
> **部署模式**：单机 / 多机（consistent hashing），RDMA（libibverbs）/ UDP（sendto/recvfrom），四种组合统一架构。

---

## 零、一句话定位

**kvspace-rdma 是一个通用 KV 存储服务器**：
- 对外暴露 `KVSpace` 接口语义（Get/Set/Del/List/Watch/Notify/Link/Unlink）
- 双传输层：InfiniBand/RoCE RDMA（RC 队列对）+ UDP（消息语义，seq/retry 可靠性）
- 单机直连或多机分片（consistent hashing key→node 路由）
- 目标：P99 GET latency < 5 μs（RDMA）/ < 50 μs（UDP），单核吞吐 > 5 Mops/s（RDMA）

### 四种部署模式

| 模式 | 传输 | 路由 | 适用场景 |
|------|------|------|---------|
| 单机 RDMA | libibverbs RC | 直连 | 有 HCA 的单机极低延迟 |
| 单机 UDP | sendto/recvfrom | 直连 | 无 RDMA 硬件，本机或同机架 |
| 多机 RDMA | libibverbs RC | consistent hashing | 有 HCA 的集群 |
| 多机 UDP | sendto/recvfrom | consistent hashing | 无 RDMA 的集群 |

全部四种模式共享同一 wire protocol、同一 store 层、同一 server 调度逻辑。

## 一、行业对标（必须超过或持平）

| 系统 | 年份 | 核心贡献 | kvspace-rdma 借鉴点 |
|------|------|---------|-------------------|
| **Pilaf** (ATC'13) | 2013 | 客户端单边 RDMA READ 直接查 cuckoo hash，GET 零 server CPU | 服务端内存布局对 one-sided 友好的 cuckoo hash |
| **HERD** (SIGCOMM'14) | 2014 | RDMA WRITE 发送请求，UD 组播回复，揭示 QP 数量瓶颈 | 用 WRITE 发送请求（绕过 RECV 配额），控制 QP 数量 |
| **FaRM** (NSDI'14) | 2014 | one-sided RDMA READ + distributed tx；循环缓冲区轮询 | 无锁环形 RX 缓冲区，CQ 忙轮询代替中断 |
| **DrTM** (SOSP'15) | 2015 | HTM + RDMA 组合事务，latency < 1 μs | 热路径无锁（HTM 或 lock-free CAS） |
| **Clover** (ATC'20) | 2020 | 解耦元数据与数据，cold-tier 直接 RDMA | Link/List 元数据路径独立，不影响 Get/Set 热路径 |
| **MICA** (NSDI'14) | 2014 | DPDK 每核独立分片，消除跨核通信 | 多核分片：key hash 决定 owner core，零跨核通信 |

**与以上系统的关键差异**：
1. 实现完整的 `Watch`（BLPOP 语义）和 `Notify`（LPUSH 语义），上述系统均未覆盖。
2. 支持 `Link`/`Unlink`（路径软链接），元数据操作均为 two-sided（server CPU 参与）。
3. 统一 RDMA/UDP 双传输，同一套 store 层不感知传输方式。
4. single-node 和 multi-node 共用接口，部署时可选择。

## 二、接口语义契约

```
Get(key)             → (value, error)         — 单 key 精确查找
Set(key, value)      → error                  — 写入并维护目录索引
Del(keys...)         → error                  — 精确删除（含索引清理）
GetMany(keys)        → ([]value, error)       — 多 key，缺失位置为 Null
SetMany(pairs)       → error                  — 批量写入
List(prefix)         → ([]string, error)      — 列出直接子项名（一级）
DelTree(prefix)      → error                  — 递归删除；prefix 是链接则只删链接本身
Watch(key, timeout)  → (value, error)         — 阻塞等待，BLPOP 语义
Notify(key, value)   → error                  — 推送，LPUSH 语义
Link(target, link)   → error                  — 软链接：link → target
Unlink(link)         → error                  — 删除链接本身，不影响 target
DisConn()            → error                  — 客户端主动断开
```

```
└───────┬──────────┘
        │
  ┌─────┴─────┐
  │           │
┌─▼──────┐ ┌──▼─────┐
│ RDMA   │ │  UDP   │
│ RC S/R │ │ sendto │
└────────┘ └────────┘
        │
┌───────▼──────────┐
│  Server Core     │
│  ┌─────────────┐ │
│  │ 元存 store/  │ │  cuckoo + dir + watch + symlink
│  │ DataRef →   │ │  slot 存储 inline val 或 DataRef
│  └──────┬──────┘ │
│         │        │
│  ┌──────▼──────┐ │
│  │ 扩展存储     │ │  slab (CPU) / GPU VRAM / RDMA / NVMe
│  │ ext/        │ │  统一 ExtStorage 接口
│  └─────────────┘ │
└──────────────────┘
```

**分层规则（严格单向依赖）**：
```
server/ → transport/  (或 router/)
        → store/      (元存)
        → ext/        (扩展存储)
        → proto/

store/  → ext/       (通过 DataRef 引用，不直接访问)
        → （仅 STL + atomic）

ext/    → （各后端独立：GPU/CUDA, slab, NVMe io_uring）
proto/  → （仅 cstdint）
transport/ → proto/  (frame 编解码)
```

- `store/` 层**禁止** include 任何 `transport/` 或 `server/` 头文件
- `store/` 不直接访问扩展存储，只管理 DataRef。实际読み書き由 `server/handlers` 协调 store 和 ext
- `ext/` 层**禁止** include `store/`、`transport/`、`server/`
- `server/handlers.h` 是唯一允许同时依赖 `store/` 和 `ext/` 的位置

## 四、传输层

### 4.1 RDMA Transport

同原设计（3.1–3.4 节保持不变）：RC QP，WRITE 发请求 + SEND 应答，per-core CQ 忙轮询。

### 4.2 UDP Transport

UDP 天然是**消息语义**（一个 datagram = 一条消息），与 RDMA SEND/RECV 完全对齐。

**为什么不用 TCP**：
- TCP 是字节流，需额外做帧切分（length prefix + buffering + 处理半包/粘包）
- 小 KV（几十～几百字节）远小于 MTU，无分片
- 数据中心丢包率 < 0.001%，UDP 的"不可靠"在此场景可忽略

**可靠性：seq + 超时重试**

```cpp
struct UdpHeader {
    uint32_t seq;     // client 单调递增，server 原样返回
    uint8_t  retry;   // 0 = 首次，1 = 重试
    uint8_t  reserved[3];
};
// 后接 standard wire protocol frame（32B header + payload）
```

逻辑：
1. Client `sendto(req)`，启动超时定时器（默认 1ms）
2. 超时前收到 `seq` 匹配的响应 → 成功
3. 超时 → retry=1 重发，间隔翻倍（backoff），最多 3 次
4. Server 收到 retry=1 的请求 → 幂等执行或忽略重复（通过 seq 去重）

**server 端**：单 `socket()` + `recvfrom()` 多路复用，无需 per-client fd。零连接状态，纯 stateless。

**UDP vs RDMA 一致性**：

| | UDP | RDMA SEND |
|--|-----|-----------|
| 消息边界 | ✅ datagram | ✅ WR |
| 帧切分 | ❌ 不需要 | ❌ 不需要 |
| 连接状态 | ❌ 无 | QP per client |
| 延迟 | ~10–50 μs | ~1–5 μs |
| 依赖 | 无 | libibverbs + HCA |

### 4.3 Wire Protocol（帧格式）

所有消息均为**定长 header + 变长 payload**，header 恰好 32 bytes：

```
┌────────────────────────────────────────────────────────────────┐
│ magic:u16=0x4B56 | version:u8=1 | op:u8 | seq:u32             │ 8B
│ key_len:u16      | val_len:u32  | flags:u16                    │ 8B
│ reserved[16]                                                   │ 16B
├────────────────────────────────────────────────────────────────┤
│ key_data[key_len]                                              │
│ val_data[val_len]   (Set/Notify 才有)                           │
└────────────────────────────────────────────────────────────────┘

op 编码：
  0x01 Get     0x02 GetMany  0x03 Set     0x04 SetMany
  0x05 Del     0x06 DelTree  0x07 List
  0x08 Watch   0x09 Notify
  0x0A Link    0x0B Unlink   0x0C DisConn
  0x81 Resp_OK 0x82 Resp_Err 0x83 Resp_Push（Watch server-push）

flags（Request）：
  bit0 = TAIL_CALL（DisConn 时置位）
  bit1 = BATCH_MORE（SetMany/GetMany 批次未结束）

flags（Response）：
  bit0 = HAS_MORE（GetMany/List 多值时分片）
  bit1 = TIMEOUT（Watch 超时）
```

## 五、存储架构：元存 / 扩展存储

kvspace 分为两层：**元存**负责所有 key 的寻址，**扩展存储**负责大 value 和 tensor 数据的实际存放。

### 5.1 核心原则

```
元存 (Meta Storage) = kvspace 本身
  └── 存储内容：key → 路由信息
      - 小 value（≤ 24B）：slot 内 inline 存储
      - 大 value（> 24B）或 tensor：slot 内存 DataRef（指向扩展存储）
      - tensor 的 shape、dtype：放在元存（在 slot inline 或 DataRef 指向的大小中）
      - 目录索引、Watch/Notify 队列、软链接

扩展存储 (Extension Storage) = 独立存储后端
  └── 存储内容：实际数据
      - 大 value 的 raw bytes
      - tensor 的 data buffer（GPU 显存、RDMA Hugepage、NVMe SSD 等）
```

**分离依据**：不是按 XValue 类型一刀切，而是按**数据的物理位置**。
- 判断条件：
  1. `value bytes > 24`（slot inline 装不下） → 扩展存储
  2. `XValue.Kind() == "tensor"` → tensor data 始终在扩展存储，tensor metadata（shape、dtype、DataRef）在元存

### 5.2 Slot 布局（64 bytes，不变）

```
┌────────────────────────────────────────────────────────────────┐
│ version:u32 | flags:u8 | key_len:u16 | val_meta_len:u16       │ 8B
│ key[24]                                                        │ 24B
│ DataRef:u64  (或 inline_val ≤ 24B 时存 inline bytes)          │ 8B
│ reserved[24]                                                   │ 24B
├────────────────────────────────────────────────────────────────┤
│                       总计：64 bytes                            │
└────────────────────────────────────────────────────────────────┘

flags（1 byte）：
  bit0 = INLINE      (1 = inline value ≤ 24B, DataRef 字段存储实际数据)
  bit1 = TENSOR      (1 = value 是 tensor，DataRef 指向扩展存储)
  bit2 = LARGE_VAL   (1 = value > 24B 且非 tensor，DataRef 指向扩展存储)
```

- `INLINE`：val_meta_len 是实际 value 长度（≤ 24），DataRef 字段存 inline bytes
- `LARGE_VAL`：DataRef 是指向扩展存储的引用，val_meta_len 是 value 总长度
- `TENSOR`：shape + dtype 存在 key[24] 之外的扩展区或 DataRef 内，tensor data 在扩展存储

### 5.3 DataRef 结构（8 bytes）

```
┌────────────────────────────────────────────────────────────────┐
│ storage_id:u8 | offset:u56                                     │ 8B
└────────────────────────────────────────────────────────────────┘

storage_id 编码：
  0x00   空 / 未分配
  0x01   CPU slab heap（当前 kvspace-rdma 默认）
  0x02   GPU VRAM（CUDA / OpenCL）
  0x03   RDMA Hugepage
  0x04   NVMe SSD（io_uring）
  0x10–  自定义扩展
```

### 5.4 扩展存储接口

```cpp
// 扩展存储后端统一接口
class ExtStorage {
public:
    virtual uint8_t storage_id() const = 0;

    // 分配空间，返回 offset
    virtual uint64_t alloc(size_t size) = 0;
    // 释放
    virtual void     free(uint64_t offset, size_t size) = 0;

    // 读写
    virtual void     read(uint64_t offset, void* buf, size_t size) = 0;
    virtual void     write(uint64_t offset, const void* data, size_t size) = 0;

    // tensor 专用：异步拷贝（GPU → RDMA / GPU → CPU）
    virtual void     copy_async(uint64_t dst_offset, uint64_t src_offset,
                                size_t size, int stream) {}
    virtual void     sync(int stream) {}
};
```

### 5.5 Get/Set 路径（含 tensor）

```
Set("/tensor/x", tensor(float32, [1024, 768], data_ptr)):
  
  1. XValue.Kind() == "tensor" → 分离 metadata 和 data
  2. DataRef ref = ext->alloc(data_size)             ← data 写入扩展存储（GPU VRAM）
     ext->write(ref.offset, data_ptr, data_size)
  3. meta = encode(shape, dtype, ref)                 ← 元数据（shape+dtype+DataRef）
  4. slot.Set("/tensor/x", meta)                      ← 元存：cuckoo hash

Get("/tensor/x"):
  
  1. meta = slot.Get("/tensor/x")                     ← 元存读取
  2. ref = decode_data_ref(meta)                      ← 提取 DataRef
  3. data = ext->read(ref.offset, data_size)          ← 扩展存储读取
  4. return XValue::Tensor(shape, dtype, data)
```

**注意**：kvspace server 本身不解析 tensor shape/dtype 语义。metadata 的编解码由 client（kvspace-cpp）负责。server 只看到 opaque bytes，根据 flags 决定路由到 inline 还是 ext storage。

### 5.2 Watch Queue：MPSC 链表

```cpp
struct WatchQueue {
    atomic<ValueNode*> head;       // Notify push 到这里
    atomic<Watcher*>   waiters;    // 等待链表
};
```

- `Notify(key, val)` = push value node → CAS 摘下 waiters 链表 → 逐个唤醒
- `Watch(key, timeout)` = 尝试 pop → 失败则注册 watcher → 阻塞等信号

## 六、并发模型

### 6.1 Per-core Shard（MICA 方案）

```
Core 0           Core 1           Core 2           Core 3
  │                │                │                │
  ▼                ▼                ▼                ▼
Shard[0]        Shard[1]        Shard[2]        Shard[3]
(hash[0,4,8…])  (hash[1,5,9…])  (hash[2,6,10…])  (hash[3,7,11…])
  │                │                │                │
  ▼                ▼                ▼                ▼
CQ/UDP poller    poller          poller           poller
```

- `key → shard` = `hash(key) % num_shards`
- 每个 shard 独立的 cuckoo table 分区，**零跨核通信**，零 lock contention
- `List`/`DelTree`（跨 shard 操作）：broadcast 到所有 shard，聚合结果

### 6.2 CQ/UDP 轮询策略

```cpp
// 热路径：忙轮询
while (true) {
    int n = ibv_poll_cq(cq, BATCH, wcs);     // RDMA
    // 或
    int n = recvfrom(sock, buf, BUF_SZ, ...); // UDP
    if (n > 0) dispatch(buf, n);
}
```

- 生产模式：每核专用轮询线程，`sched_setaffinity` 绑核
- 调试模式：允许 `epoll` + `recvfrom`（UDP）

## 七、多机路由

### 7.1 Consistent Hashing Ring

```
       ┌───────────────────────────────┐
       │         hash ring             │
       │  0 ─── node_0 ─── 2^32/3     │
       │  2^32/3 ─── node_1 ─── 2^64/3│
       │  2^64/3 ─── node_0(副本) ── 2^32│
       └───────────────────────────────┘

key "hello" → hash("hello") = 0x1A2B... → 落在 node_0 → 路由到 node_0
```

- 虚拟节点 256 个/物理节点，均匀分布
- 路由信息通过静态配置或 gossip 传播（v0 静态配置）

### 7.2 路由策略

| 操作 | 路由方式 |
|------|---------|
| Get/Set/Del | 单 key，路由到 owner node |
| GetMany/SetMany | per-key 按 node 分组，并行请求 |
| List/DelTree | broadcast 所有节点，聚合结果 |
| Watch/Notify | Watch 注册在 key owner node；跨节点时 client 做 relay |
| Link/Unlink | 路由到 linkpath 所在 node |

### 7.3 Cross-node 请求处理

```
Client → node_A (router: key belongs to node_B)
           │
           ├── 转发请求到 node_B
           ├── node_B 处理，返回结果
           └── node_A 返回结果给 client
```

Client 更新本地路由表（`X-Redirect: node_B` header），后续直接连 node_B。

**单向依赖（严格）**：
```
main.cc → server/ → transport/ (或 router/)
                  → store/
                  → proto/

store/  → （无内部依赖，只用 STL + 锁原语）
proto/  → （只用 cstdint）
transport/ → proto/
```

- `store/` 禁止 include `transport/` 或 `server/`
- `transport/` 禁止 include `store/` 或 `server/`
- `server/handlers.h` 是唯一允许同时依赖 `transport/` 和 `store/` 的位置

## 十、性能目标

| 指标 | RDMA 目标 | UDP 目标 | 条件 |
|------|----------|---------|------|
| GET P50 latency | < 2 μs | < 20 μs | 4B key, inline val, 1 client |
| GET P99 latency | < 5 μs | < 50 μs | 同上 |
| SET P99 latency | < 8 μs | < 60 μs | 4B key/val, 1 client |
| 单核 GET 吞吐 | > 5 Mops/s | > 500 Kops/s | 4 core, 100% read |
| 连接扩展 | 1000 QP | 无限制（stateless） | latency ≤ 1.5× baseline |

## 十一、禁止项（不得妥协）

| 编号 | 禁止 | 理由 |
|------|------|------|
| P1 | 热路径 malloc/free | slab/pool only，每次 malloc ≈ 100ns |
| P2 | 热路径 mutex lock | lock-free CAS 或 seqlock；mutex 引入 kernel syscall |
| P3 | 热路径中断模式 | 一律忙轮询；中断 latency > 50 μs |
| P4 | store/ 层感知传输方式 | store 只操作 `void*`/`vector<uint8_t>` |
| P5 | 跨 shard 同步调用 | 死锁风险；用消息传递 |
| P6 | 函数超 50 行或文件超 500 行 | 超限必须拆分 |
| P7 | 忽略 I/O 返回值 | 每个 send/recv/poll 结果必须检查 |
| P8 | UDP transport 使用 TCP | UDP 是消息语义，TCP 字节流需额外帧切分层 |
| P9 | store/ 层直接访问扩展存储 | store 只管理 DataRef，实际 I/O 由 handlers 层协调 ext/ |
| P10 | ext/ 层感知 KV 语义（key、directory、link） | ext 只理解 offset+size，不知道 key |
| P11 | tensor data 存入元存 slot | tensor data 始终走 DataRef → ext；元存只放 shape+dtype+DataRef |
